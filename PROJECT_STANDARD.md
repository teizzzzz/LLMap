# mapv2.0 项目标准

更新时间：2026-06-15

## 1. 项目定位

`mapv2.0` 是一个言语景观采集、记录、AI 预标注、人工复核与地图展示工具。项目参考 LingScape / linguistic landscape fieldwork 类工具的基本思想：以现场照片为入口，将每一个可观察的 sign、看板、贴纸、路面文字、临时告示、车辆文字等记录为结构化数据，并保留照片、地理位置、采集时间、语言属性和研究者备注。

当前项目形态是轻量本地工具包：

- Python Tkinter GUI：用于照片重命名、EXIF 提取、CSV 生成、AI 批量判定、精度测试、启动器。
- HTML 静态工具：用于地图展示、图片裁剪、人工复核。
- CSV：作为核心数据存储格式。
- `photos/`：作为图片资产目录。

本项目当前不是完整后端系统，也没有数据库；所有标准都应优先保证 CSV 与本地文件流转稳定。

## 2. 当前文件职责

| 文件/目录 | 职责 | 标准说明 |
|---|---|---|
| `启动.bat` | Windows 一键启动本地 HTTP 服务并打开地图 | 服务端口默认为 `8080`，入口为 `index.html` |
| `launcher.py` | ASCII 启动入口 | 包装调用 `ランチャー.py`，避免 Windows batch 直接引用日文文件名失败 |
| `ランチャー.py` | 图形化启动器 | 管理本地服务器，打开地图、裁剪、复核、重命名、合并、AI 工具 |
| `ll_rename_gui.py` | 照片重命名与初始 CSV 生成 | 读取 EXIF 时间/GPS，输出 `photos/`、`thumbs/`、`data.csv` |
| `crop_tool.html` | 单张照片 sign 裁剪工具 | 从全景图裁出多个 sign 图片，并导出 crop CSV 行 |
| `csv_merge.py` | CSV 合并工具 | 删除已有裁剪图的全景行，追加 crop CSV，重排 `ImageID` |
| `ll_ai_judge_gui.py` | Python AI 批量判定工具 | 读取 CSV 与照片，调用 Claude/Gemini 写入 AI 字段 |
| `review_tool_v2.html` | 人工复核工具 | 读取 CSV 和照片，编辑字段，管理 `_status`，导出复核 CSV |
| `index.html` | 地图展示页 | 读取 `data.csv`，用 Leaflet 展示点位与详情 |
| `accuracy_test_gui.py` | AI 图片压缩/准确率测试工具 | 对比原寸、推荐、轻量、超轻量配置的 token、速度、字段一致性 |
| `data.csv` | 当前主数据 | 地图默认读取此文件 |
| `data_ai.csv` | AI 判定结果数据 | AI 工具优先读取此文件 |
| `test_csv/` | 测试与实验数据 | 保存未判定样例、AI 判定样例、压缩精度测试结果 |
| `photos/` | 示例照片 | 文件名必须与 CSV 的 `写真ファイル名` 对齐 |
| `mapv2.0.zip` | 项目压缩包 | 不作为运行时依赖 |

## 3. 标准工作流

1. 现场采集照片。
2. 使用 `ll_rename_gui.py` 批量重命名，生成标准照片名和初始 `data.csv`。
3. 如一张照片包含多个 sign，使用 `crop_tool.html` 裁剪，导出裁剪图片与 crop CSV。
4. 使用 `csv_merge.py` 将 crop CSV 合并回主 CSV，并删除已有裁剪图的全景行。
5. 使用 `ll_ai_judge_gui.py` 或 `review_tool_v2.html` 做 AI 预标注。
6. 使用 `review_tool_v2.html` 人工确认、修正、承认。
7. 将最终确认后的 CSV 保存为主 `data.csv` 或明确版本名。
8. 使用 `index.html` 在地图上展示最终数据。

AI 结果只作为预标注；进入展示或论文分析前，必须经过人工复核并将 `_status` 改为 `reviewed`。

推荐项目结构：

```text
mapv2.0/

  csv_merge.py
  启动.bat
  data.csv
  data_ai.csv
  photos/
  thumbs/

```

运行数据与实验数据应分开：

- `data.csv`：地图默认展示用数据。
- `data_ai.csv`：AI 预标注中间结果。
- `LL_reviewed_YYYY-MM-DD.csv`：人工复核导出结果。
- `test_csv/`：只放测试、实验、模型比较结果，不作为正式地图默认数据。

## 5. 命名标准

### 5.1 地区代码

地区代码使用 3 位大写英文字母。

已有预设：

- `HAS`：橋本
- `YKH`：横浜中華街
- `SKO`：新大久保

新增地区必须先登记地区代码，避免不同采集批次冲突。

### 5.2 照片 ID

标准格式：

```text
AREA-YYYYMMDD-NNN
```

示例：

```text
HAS-20260531-010
```

规则：

- `AREA` 为 3 位地区代码。
- `YYYYMMDD` 取照片 EXIF 拍摄日期；无 EXIF 时使用文件修改时间，并在日志中提示。
- `NNN` 为同一批次按时间排序后的 3 位序号，从 `001` 开始。

### 5.3 照片文件名

全景/原始标准照片：

```text
HAS-20260531-010.jpg
```

裁剪 sign 照片：

```text
HAS-20260531-010-a.jpg
HAS-20260531-010-b.jpg
```

裁剪后，`写真ID` 仍应保留原照片 ID，例如 `HAS-20260531-010`；裁剪图文件名写入 `写真ファイル名`；原全景文件名写入 `全景写真ファイル名`。

### 5.4 SignID

标准格式：

```text
AREA-SNNN-suffix
```

示例：

```text
HAS-S010-a
HAS-S010-b
```

规则：

- `SNNN` 对应照片序号。
- `suffix` 表示同一照片中的第几个 sign，推荐 `a, b, c...`。
- 单张照片只有一个 sign 时也使用 `a`，保证后续可扩展。

### 5.5 ImageID

`ImageID` 当前只在部分工具中生成，格式为：

```text
AREA00001
```

例如：

```text
HAS00001
```

为了兼容现有数据，`ImageID` 可作为可选字段；如果后续需要数据库化或跨批次唯一索引，应将它升级为必填字段。

## 6. CSV 字段标准

### 6.1 推荐字段顺序

正式数据建议使用以下顺序。当前工具对字段顺序有一定容忍度，但统一顺序可以减少合并和复核成本。

```csv
SignID,_status,写真ID,写真ファイル名,全景写真ファイル名,緯度,経度,採集日時,名称,主要言語,補助言語,装飾的言語,設置主体,担体類型,時間性,可読性,来源類型,視認等級,位置描述,多言語関係,想定受容者,制作品質,備考
```

扩展字段：

```csv
ImageID,言説類型,状態,層状性
```

如果使用 `ll_rename_gui.py` 或 `ll_ai_judge_gui.py`，扩展字段可能出现在 CSV 中；地图和复核工具当前主要依赖核心字段。

### 6.2 字段定义

| 字段 | 必填 | 说明 |
|---|---:|---|
| `SignID` | 是 | sign 唯一标识 |
| `_status` | 是 | 处理状态：`pending`、`ai`、`modified`、`reviewed` |
| `写真ID` | 是 | 原照片 ID |
| `写真ファイル名` | 是 | 当前记录对应的照片文件名 |
| `全景写真ファイル名` | 否 | 裁剪图对应的原全景照片 |
| `緯度` | 地图展示必填 | 十进制度纬度 |
| `経度` | 地图展示必填 | 十进制度经度 |
| `採集日時` | 建议必填 | `YYYY-MM-DD HH:MM:SS` |
| `名称` | 复核后必填 | sign 内容简名 |
| `主要言語` | 复核后必填 | sign 的主导语言 |
| `補助言語` | 否 | 辅助语言，没有则用 `—` |
| `装飾的言語` | 否 | 装饰性语言或品牌语言 |
| `設置主体` | 复核后必填 | 标牌设置主体 |
| `担体類型` | 复核后必填 | 标牌载体类型 |
| `時間性` | 人工确认 | 常设/临时等 |
| `可読性` | 人工确认 | 可读程度 |
| `来源類型` | 复核后必填 | top-down / bottom-up |
| `視認等級` | 人工确认 | 观察距离等级 |
| `位置描述` | 人工确认 | 现场位置描述 |
| `多言語関係` | 复核后必填 | 多语言之间的功能关系 |
| `想定受容者` | 人工确认 | 目标受众 |
| `制作品質` | 人工确认 | 制作质量 |
| `備考` | 否 | 研究者备注 |

### 6.3 状态流转

| 状态 | 含义 | 进入方式 |
|---|---|---|
| `pending` | 未判定/未处理 | 初始 CSV、裁剪导出 |
| `ai` | AI 已预标注 | AI 工具写入 |
| `modified` | 人工已修改 | 复核工具编辑字段 |
| `reviewed` | 人工已确认 | 复核工具点击承认 |

正式分析和地图发布优先使用 `reviewed` 数据。

## 7. 分类值标准

### 7.1 语言字段

`主要言語`、`補助言語`：

- `日本語`
- `中文`
- `韓国語`
- `英語`
- `その他`
- `—`

`装飾的言語`：

- `英語(ブランド名)`
- `英語(装飾)`
- `その他`
- `—`

### 7.2 设置主体

- `行政/公共機関`
- `商業`
- `個人/非正式`
- `不明`

### 7.3 载体类型

- `建物固定(看板)`
- `建物固定(壁画)`
- `建物固定(門牌)`
- `移動物体(停車中)`
- `貼付物(ポスター)`
- `貼付物(ステッカー)`
- `路面固定`
- `デジタル表示(LED)`
- `その他`

### 7.4 时间性

- `常設`
- `臨時/季節的`
- `不明`

### 7.5 可读性

- `完全可読`
- `部分可読`
- `不可読`

### 7.6 来源类型

- `top-down(標準化)`
- `bottom-up(ローカル自作)`
- `不明`

### 7.7 视认等级

- `歩行可読(<5m)`
- `近距可読(5-15m)`
- `遠距可読(>15m)`

### 7.8 多语言关系

- `平行型`：不同语言传达基本相同信息。
- `補完型`：不同语言提供互补信息。
- `象徴型`：某语言主要承担象征、品牌、氛围功能。
- `単言語型`：只有一种主要语言。

### 7.9 预设受众

- `本地住民`
- `特定コミュニティ`
- `観光客・外国人`
- `不明`

### 7.10 制作质量

- `専門印刷`
- `半専門(印刷)`
- `手書き`

## 8. AI 判定标准

AI 可以填写：

- `名称`
- `主要言語`
- `補助言語`
- `装飾的言語`
- `設置主体`
- `担体類型`
- `来源類型`
- `多言語関係`
- `備考`

AI 可辅助但不应最终决定：

- `時間性`
- `可読性`
- `視認等級`
- `想定受容者`
- `制作品質`
- `位置描述`

原因：这些字段依赖现场距离、观察角度、采集者路线、空间位置和研究判断，只看单张图片容易误判。

当前代码中使用或预设的模型：

- Claude：`claude-haiku-4-5-20251001`
- Gemini：`gemini-2.5-flash`
- DeepSeek：`deepseek-chat`，仅在 `review_tool_v2.html` 中出现，启用前需确认当前 API 是否支持图片输入。

API key 标准：

- 不写入 CSV。
- 不提交到项目目录。
- 不硬编码进 HTML/Python。
- 浏览器工具当前只保存在页面内存中；关闭页面后失效。

## 9. 图片处理标准

重命名阶段：

- 输出格式统一为 JPEG。
- `ll_rename_gui.py` 当前保存质量为 `92`。
- 缩略图最长边不超过 `800px`，质量为 `80`。
- EXIF Orientation 应被应用，避免竖图横置。

AI 判定阶段：

- 推荐最大边 `1024px`、JPEG 质量 `75`。
- 对 Claude，压缩通常可减少 token。
- 对 Gemini，当前测试工具认为图片 token 近似固定，压缩主要影响上传速度和前端加载。

裁剪阶段：

- 裁剪图保存质量当前为 `0.92`。
- 裁剪 ID 使用全景 `写真ID` 加 suffix。
- 裁剪后应保留全景文件名，便于追溯。
- `crop_tool.html` 支持单张图片导入，也支持通过文件夹导入批量图片，并用上一张/下一张导航。

## 11. 复核标准

人工复核时至少检查：

- 照片是否能打开。
- `SignID` 与照片文件是否匹配。
- 坐标是否明显偏离采集区域。
- `名称` 是否是可理解的 sign 简名，而不是长篇描述。
- `主要言語` 是否基于 sign 中最核心、最醒目的语言。
- `補助言語` 和 `装飾的言語` 是否与主语言区分清楚。
- `来源類型` 是否符合研究判断。
- `備考` 是否记录了 AI 无法稳定判断但研究上重要的信息。

复核完成后，将 `_status` 改为 `reviewed`。

## 12. 测试与验证标准

Python 语法检查：

```powershell
python -m py_compile ランチャー.py ll_rename_gui.py csv_merge.py ll_ai_judge_gui.py accuracy_test_gui.py
```

本地服务器：

```powershell
python -m http.server 8080
```

浏览器访问：

```text
http://localhost:8080/index.html
http://localhost:8080/crop_tool.html
http://localhost:8080/review_tool_v2.html
```

CSV 检查：

- 使用 `utf-8-sig` 编码，兼容 Excel。
- 表头必须包含 `SignID`、`写真ID`、`写真ファイル名`、`緯度`、`経度`、`採集日時`、`_status`。
- 多行文本、逗号、双引号必须正确 CSV 转义。

AI 精度测试：

- 使用 `accuracy_test_gui.py`。
- 保存结果到 `test_csv/`。
- 至少对比原寸、推荐、轻量、超轻量四档。
- 记录 token、耗时、KB、核心字段一致率。

## 13. 当前状态与已知风险

本轮已修复：

- `启动.bat` 现在优先打开 `ランチャー.py`，不再重复启动同一端口的 `http.server`。
- `ll_ai_judge_gui.py` 已补上 `build_prompt(fields)`，字段选择不会再触发未定义函数错误。
- `ll_ai_judge_gui.py` 的 `image_to_base64()` 已支持 `max_size` 与 `quality` 参数，AI 批处理压缩设置可生效。
- `csv_merge.py` 会从全景行继承裁剪行的 `緯度`、`経度`、`採集日時`、`全景写真ファイル名`。
- `index.html` 详情编号优先使用 `SignID` / `ImageID`，并支持手动打开 CSV。
- `crop_tool.html` 的 CSV 导出换行与引号转义已修复。
- `review_tool_v2.html` 的 CSV 解析、导出表头和新增 `SignID` 逻辑已加强。

仍需注意：

- `review_tool_v2.html` 的浏览器端 AI API 调用可能受 CORS、模型视觉能力或 provider 支持限制影响；稳定批处理优先使用 Python GUI。
- 根目录 `data.csv` 引用了大量 `photos/` 中不存在的图片，地图详情会显示占位图。
- 字段顺序在旧 CSV、测试 CSV 和工具导出之间仍可能不完全一致；工具已尽量兼容，但正式数据建议按本文档标准表头整理。
- `mapv2.0` 当前不是 Git 仓库，版本追踪和回滚能力不足。

## 14. 后续开发原则

- CSV 是当前系统契约，新增字段前先更新本文档。
- HTML 工具应优先读取同一套字段，不各自发明表头。
- AI 字段和人工字段要分层，避免 AI 覆盖现场判断。
- 所有照片、裁剪图、CSV 都必须能通过 `SignID`、`写真ID`、`写真ファイル名` 互相追溯。
- 展示页只展示已复核数据时，应过滤 `_status == reviewed`，或在 UI 中明确标注未复核状态。
- 后续如引入数据库，仍应保留 CSV 导入导出，作为研究资料交换格式。
